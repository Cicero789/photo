1. Client gallery/photo APIs do not match db/schema.sql
Priority: P0

Why this is likely breaking the app
The D1 schema in db/schema.sql defines:
- client_galleries: id, client_site_id, name, created_at, updated_at, deleted_at
- client_gallery_photos: id, gallery_id, storage_key, filename, width, height, file_size, sort_order, created_at, deleted_at

But several endpoints use columns that do not exist:
- functions/api/clients/[id]/photos.ts uses client_galleries.slug.
- functions/api/clients/[id]/photos.ts inserts client_gallery_photos.caption and uploaded_at.
- functions/api/clients/[id]/photos.ts does not insert the required client_gallery_photos.id.
- functions/api/clients/[id]/galleries/[gid].ts also inserts caption/uploaded_at and omits id.
- functions/api/clients/public/[slug]/galleries/[gid].ts selects caption/uploaded_at.

Expected symptoms
- Client photo uploads fail with D1 errors such as "no such column: slug", "no such column: caption", "no such column: uploaded_at", or "NOT NULL constraint failed: client_gallery_photos.id".
- Public client galleries can load with missing file names or fail completely.

Recommended direction
Update the functions to match the schema. Do not add new columns unless you truly want slug/caption/uploaded_at long term.

```ts
async function getNextSortOrder(db: D1Database, galleryId: string): Promise<number> {
```

2. Dashboard broken image on "games" block
Priority: P0/P1

What I found locally
There is no current source component or route named game/games in src, functions, public, db, README, ARCHITECTURE, or AUDIT_REPORT.

The template thumbnails used by the photographer dashboard are rendered in:
- src/components/photographer/TemplatePicker.tsx

The registry is in:
- src/components/templates/types.ts

I compared every registered template id against public/thumbnails/*.jpg:
- Registered templates: 263.
- Missing registered thumbnails: 0.

So the broken image on production is probably one of these:
- Production deploy is stale and missing an image that exists locally.
- A saved user/profile/template value still points to an old id such as "clean-minimal".
- A custom ad tile or dashboard tile points to an external image URL that the CSP blocks.
- A database value contains a bad or deleted image URL.

Fix old template id fallback first
Several places still use "clean-minimal", but the current registry id appears to be "1-clean".

Affected files:
- src/pages/DashboardPage.tsx
- src/pages/PhotographerProfilePage.tsx
- src/components/photographer/TemplatePicker.tsx

Add a central normalizer:

```ts
// src/lib/templateIds.ts
import { TEMPLATE_REGISTRY } from "../components/templates/types";

export const DEFAULT_TEMPLATE_ID = "1-clean";

const TEMPLATE_ID_ALIASES: Record<string, string> = {
  "clean-minimal": "1-clean",
  "cinematic-dark": "2-cinematic",
  "editorial-magazine": "3-editorial",
  "instagram-grid": "4-instagram",
  "masonry-wall": "5-masonry",
  "split-hero": "6-split",
  "vertical-scroll": "7-vertical",
  "carousel-spotlight": "8-carousel",
  "story-cards": "9-story-cards",
  "brutalist-bold": "10-brutalist",
};

const TEMPLATE_IDS = new Set(TEMPLATE_REGISTRY.map((template) => template.id));

export function normalizeTemplateId(value?: string | null): string {
  if (!value) return DEFAULT_TEMPLATE_ID;
  const resolved = TEMPLATE_ID_ALIASES[value] ?? value;
  return TEMPLATE_IDS.has(resolved) ? resolved : DEFAULT_TEMPLATE_ID;
}
```

```ts
  ...,
  template: normalizeTemplateId(design.template),
});

  onSelectTemplate={handleTemplateChange}
/>
```

If the "games" block is actually an ad tile
The CSP currently only allows images from self, data, blob, Unsplash, and Mapbox. Admin ad tiles allow arbitrary image URLs. That means an ad image hosted elsewhere will be blocked by the browser and look broken.

Recommended safe fix:
- Store ad tile images in R2 and serve them through /api/media/photos/... or a dedicated /api/media/ads/... route.
- Or validate a short allow-list of image hosts at save time and add only those hosts to CSP.

Do not solve this by adding every https image host to CSP unless the business accepts that risk.


3. npm run lint fails
Priority: P1

The repo uses --max-warnings 0, so warnings are build/release blockers.

Current lint failure categories:
- @typescript-eslint/no-floating-promises
- react-hooks/exhaustive-deps
- one unused eslint-disable

Recommended approach
- Keep the strict lint rule.
- For intentional fire-and-forget operations, use void and make sure the promise handles errors.
- For hooks, use useCallback or include the actual dependency.

Example pattern:

```tsx
const fetchMapData = useCallback(async () => {
  const result = await api.get<MapData>("/api/events/map");
  setMapData(result);
}, []);

useEffect(() => {
  if (viewMode === "map") {
    void fetchMapData();
  }
}, [viewMode, fetchMapData]);
```

Example event handler:

```tsx
const handleDelete = (id: string) => {
  void deleteThing(id).catch((error) => {
    console.error("Delete failed", error);
    setError("Delete failed. Please try again.");
  });
};
```

Do not ignore no-floating-promises globally. It is catching real async UI failure cases.

6. Upload endpoints should reject oversized bodies before request.formData()
Priority: P1/P2

Cloudflare Workers can buffer formData() in memory. Several upload endpoints check file size after parsing the request. Add a content-length guard before formData().

Affected endpoints to inspect:
- functions/api/clients/[id]/photos.ts
- functions/api/clients/[id]/galleries/[gid].ts

Pattern:

```ts
import { rejectOversizedRequest } from "../../../lib/upload-policy";

const tooLarge = rejectOversizedRequest(context.request, 25 * 1024 * 1024);
if (tooLarge) return tooLarge;

const formData = await context.request.formData();
```

If the endpoint accepts multiple files, set the limit to the actual maximum total request size, not just single-file size.

8. CSP can block valid business images
Priority: P1/P2

Current middleware CSP limits images to:
- self
- data
- blob
- images.unsplash.com
- *.mapbox.com

That is good for security, but it conflicts with arbitrary ad tile image URLs and some custom content.

Recommended options
1. Best: store uploaded/custom images in R2 and serve them from your own /api/media route.
2. Good: validate image URL hosts server-side and add exactly those hosts to img-src.
3. Avoid: broad img-src https: unless absolutely necessary.

Add URL validation for ad tiles:

```ts
const ALLOWED_AD_IMAGE_HOSTS = new Set([
  "images.unsplash.com",
  "framenest.photos",
]);

function validateAdImageUrl(value: string): string {
  const url = new URL(value);
  if (url.protocol !== "https:" || !ALLOWED_AD_IMAGE_HOSTS.has(url.hostname)) {
    throw new Error("Unsupported ad image host");
  }
  return url.toString();
}
```

Then align functions/_middleware.ts CSP with the allow-list.

12. Rate limiting is duplicated on some auth paths
Priority: P3

The middleware applies rate limits to certain auth prefixes, and some auth handlers also call checkRateLimit. This can be okay, but it may create confusing buckets or double-increment behavior.

Recommended fix
- Decide which rate limits belong globally in middleware and which belong inside handlers.
- Keep stricter login/gate/password-reset rate limits, but make bucket names obvious and avoid accidental double-counting for the same action.


13. Password comparison helper can be modernized
Priority: P3

functions/lib/password.ts has a custom timingSafeEqual helper. Since stored password hashes are fixed length, this is probably acceptable, but modern Workers support stronger crypto primitives.

Recommended improvement
- Prefer Web Crypto APIs for fixed-length digest comparison where available.
- Do not introduce string equality checks for secrets/hashes.
