1. Single remaining issue before real customers upload private photos

If I have to pick one privacy blocker, I would fix private media access revocation and signing before allowing real customers to upload private photos.

Current issue:
- functions/lib/signed-url.ts signs only storageKey and expires.
- functions/lib/signed-url.ts verifies with expected === signature.
- functions/api/media/[[path]].ts lets any valid signed URL bypass normal database authorization.
- The signed URL does not include kind, so a valid signature is not explicitly bound to photos vs videos.
- The signed URL does not re-check that the row is still present, private, not deleted, and still belongs to an allowed album/event/gallery.

Why this matters:
- Private photos are only as private as the media serving path.
- A signed URL is currently a one-hour bearer credential.
- If a photo is deleted, moved, made private, or removed from an album, an already-issued signed URL can continue to work until expiry.
- The direct string compare is not ideal for HMACs.

What I would do:
- Sign v1:kind:storageKey:expires.
- Verify in constant time.
- Reduce default private media TTL to 10-15 minutes.
- On signed URL requests, still do a cheap DB existence/deleted_at/visibility check for private media.
- Add a test that a photos signature cannot fetch videos and that deleted media stops serving.

Important nuance:
If "private photos" means the client-gallery product path, there is also a functional launch blocker: client gallery endpoints still reference columns not present in db/schema.sql. That will likely 500 unless production D1 has an untracked migration. But the privacy blocker I would fix first is signed media access.

Suggested signed URL code

```ts
type MediaKind = "photos" | "videos";

function mediaPayload(kind: MediaKind, storageKey: string, expires: number): string {
  return `v1:${kind}:${storageKey}:${expires}`;
}

async function hmacHex(secret: string, data: string): Promise<string> {
  const key = await crypto.subtle.importKey(
    "raw",
    new TextEncoder().encode(secret),
    { name: "HMAC", hash: "SHA-256" },
    false,
    ["sign"]
  );
  const sig = await crypto.subtle.sign("HMAC", key, new TextEncoder().encode(data));
  return Array.from(new Uint8Array(sig))
    .map((b) => b.toString(16).padStart(2, "0"))
    .join("");
}

async function timingSafeStringEqual(a: string, b: string): Promise<boolean> {
  const encoder = new TextEncoder();
  const [aHash, bHash] = await Promise.all([
    crypto.subtle.digest("SHA-256", encoder.encode(a)),
    crypto.subtle.digest("SHA-256", encoder.encode(b)),
  ]);
  return crypto.subtle.timingSafeEqual(aHash, bHash);
}

export async function signMediaUrl(
  storageKey: string,
  secret: string,
  ttlSeconds = 900,
  kind: MediaKind = "photos"
): Promise<string> {
  const expires = Math.floor(Date.now() / 1000) + ttlSeconds;
  const sig = await hmacHex(secret, mediaPayload(kind, storageKey, expires));
  return `/api/media/${kind}/${encodeURIComponent(storageKey)}?expires=${expires}&sig=${sig}`;
}

export async function verifyMediaSignature(
  storageKey: string,
  kind: MediaKind,
  signature: string,
  expires: string,
  secret: string
): Promise<boolean> {
  const exp = Number(expires);
  if (!Number.isFinite(exp) || exp < Math.floor(Date.now() / 1000)) return false;

  const expected = await hmacHex(secret, mediaPayload(kind, storageKey, exp));
  return timingSafeStringEqual(expected, signature);
}
```

2. Architecture choice I would undo without breaking compatibility

I would undo localStorage bearer JWT auth as the primary web session model.

Current design:
- src/lib/api.ts stores photo_token in localStorage.
- Many components read localStorage directly and send Authorization: Bearer.
- JWTs embed role and space claims.
- token_version now helps, but only after a server-side version bump.

Replacement design:
- Add HttpOnly, Secure, SameSite=Lax or Strict session cookies.
- Keep Authorization bearer token support temporarily for backward compatibility.
- Store session rows or jti rows server-side for admin sessions.
- Keep token_version, but add a first-class "revoke all sessions" endpoint and UI.
- Require step-up re-auth for platform_owner destructive actions.

Why this is the design I would undo:
- It directly controls stolen-token blast radius.
- It reduces XSS impact.
- It gives you server-side revocation instead of waiting for 24-hour JWT expiry.
- It centralizes session behavior instead of every page manually touching localStorage.

Backward-compatible migration:
1. Login returns both the existing token and Set-Cookie.
2. API accepts cookie first, then Authorization header.
3. Frontend stops reading localStorage in new code.
4. After one or two releases, remove localStorage token reads from components.

4. Where this architecture breaks first at scale

It breaks on active media workflows, not raw registered user count.

If the project is on Workers Free:
- The first hard limit can be Function request volume.
- Cloudflare Workers Free has 100,000 requests/day.
- Because /api/media serves private media through a Function, photo-heavy browsing can eat that faster than ordinary page views.

If the project is on Workers Paid:
- R2 and Cloudflare edge request throughput are not the first concern.
- Cloudflare Workers scale broadly, and R2 has no egress fee.
- The first real application bottleneck is the single D1 database during bursts of writes and metadata reads.

Why D1 breaks first:
- Cloudflare documents that each individual D1 database processes queries one at a time.
- Throughput is tied to query duration: a 1 ms average can be around 1,000 qps, while a 100 ms average is around 10 qps.
- Upload workflows write one D1 row per photo/video plus event/cover metadata.
- Admin dashboards and list pages also do count queries and joined reads.

Rough practical estimate:
- 1,000 to 10,000 registered users is probably fine if daily active uploaders are low.
- 50 to 100 simultaneous uploaders, each uploading 20 to 100 files, is where D1 write queues and metadata latency become visible.
- 10,000+ daily active users browsing media-heavy private albums through /api/media can make Function request count and R2 Class B operations a cost/monitoring concern.

What I would change before that:
- Direct-to-R2 upload URLs with a finalize endpoint.
- Queue thumbnail/metadata processing.
- Batch D1 writes where safe.
- Add per-space quotas and circuit breakers.
- Cache public media aggressively and keep private media short-TTL signed.
- Consider per-tenant D1 databases or partitioning once a few large customers dominate traffic.

5. Stolen valid admin JWT: blast radius and cheapest fix

Blast radius today:
- A stolen platform_owner JWT can perform platform_owner API actions until token expiry or token_version changes.
- It can list and update users.
- It can change roles/emails/names.
- It can delete non-platform-owner users.
- It can access admin stats, orders, content tools, ads, pins, and photographer approval.
- It can read/manage events across spaces because platform_owner passes the policy helpers.
- It can likely issue or retrieve private media URLs through admin-visible event data.
- It can change public content and custom domains.
- It can reset space content through admin endpoints.

Things that currently limit blast radius:
- JWT expiry is 24 hours.
- token_version now exists and requireAuth checks it for real users.
- Password reset increments token_version.

What is missing:
- No jti denylist/session table.
- No "log out all devices" UI.
- No short-lived admin access token.
- No step-up re-auth for destructive platform_owner operations.
- Tokens live in localStorage, so a successful XSS can read them.

Cheapest fix:
Add an authenticated "revoke sessions" endpoint that increments token_version, then add a platform-owner UI button for it. Then add shorter platform_owner token TTL and step-up re-auth for dangerous actions.

Suggested endpoint

```ts
export async function onRequestPost(context: {
  request: Request;
  env: { DB?: D1Database; JWT_SECRET?: string; ENVIRONMENT?: string };
}): Promise<Response> {
  const a = await requireAuth(context.request, context.env);
  if (a instanceof Response) return a;

  await context.env.DB!.prepare(
    "UPDATE users SET token_version = token_version + 1 WHERE id = ?"
  ).bind(a.userId).run();

  return json({ success: true });
}
```

Better fix:
- Introduce a sessions table with jti, user_id, created_at, expires_at, revoked_at, device_label.
- Check jti in requireAuth.
- Store web sessions in HttpOnly cookies.
- Add re-auth for: role changes, delete user, reset space, custom domain changes, password changes, media export, billing/Stripe actions.

```ts
export function normalizeCustomDomain(input: unknown): string | null {
  if (typeof input !== "string") return null;
  const value = input.trim();
  if (!value) return null;

  let host: string;
  try {
    const url = value.includes("://") ? new URL(value) : new URL(`https://${value}`);
    host = url.hostname.toLowerCase();
  } catch {
    return null;
  }

  const labels = host.split(".");
  const valid = labels.length >= 2 && labels.every((label) =>
    label.length > 0 &&
    label.length <= 63 &&
    !label.startsWith("-") &&
    !label.endsWith("-") &&
    /^[a-z0-9-]+$/.test(label)
  );

  if (!valid) return null;
  if (host === "framenest.photos" || host.endsWith(".framenest.photos")) return null;
  return host;
}
```
