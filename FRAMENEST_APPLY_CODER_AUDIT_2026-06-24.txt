AP-002 [P0] CLIENT BLOG REQUEST/RESPONSE CONTRACT IS INCOMPATIBLE

Evidence:

  Frontend expects:
    excerpt, content, coverImageUrl, status, publishedAt

  Backend reads/returns:
    body, featuredImage, published, createdAt, updatedAt

Impact:

  Even after the URL is fixed, body content is sent under the wrong key and can
  be saved blank. Status and dates render as undefined/Invalid Date.

Recommended fix:

  Define one shared DTO and map database snake_case fields only inside the
  Function. Do not let React pages know database column names.

Sample:

  // src/contracts/client-sites.ts
  export type PublishStatus = "draft" | "published";

  export interface BlogPostDto {
    id: string;
    title: string;
    slug: string;
    excerpt: string;
    content: string;
    coverImageUrl: string | null;
    status: PublishStatus;
    publishedAt: string | null;
    createdAt: string;
    updatedAt: string;
  }

  export interface SaveBlogPostInput {
    title: string;
    slug?: string;
    excerpt?: string;
    content: string;
    coverImageUrl?: string | null;
    status?: PublishStatus;
  }

  function toBlogPostDto(row: any): BlogPostDto {
    return {
      id: row.id,
      title: row.title,
      slug: row.slug,
      excerpt: row.excerpt ?? "",
      content: row.body ?? "",
      coverImageUrl: row.featured_image ?? null,
      status: row.published === 1 ? "published" : "draft",
      publishedAt: row.published_at ?? null,
      createdAt: row.created_at,
      updatedAt: row.updated_at,
    };
  }

Database recommendation:

  Add excerpt and published_at through a forward migration. Do not overload
  created_at as the publication date.

Acceptance tests:

  - Round-trip every DTO field through create, GET, update, and GET.
  - Publishing sets published_at once.
  - Unpublishing does not destroy content.
  - Invalid status is rejected with 400.

Evidence:

  Do not maintain two separate client upload implementations. Extract one
  gallery upload service and have the photo-pool endpoint create/use a
  conventional default gallery using the actual schema.

Correct insertion shape:

  const galleryId = crypto.randomUUID();

  await db.prepare(
    `INSERT INTO client_galleries
       (id, client_site_id, name, created_at, updated_at)
     VALUES (?, ?, ?, ?, ?)`
  ).bind(galleryId, siteId, "Photos", now, now).run();

  const photoId = crypto.randomUUID();
  await db.prepare(
    `INSERT INTO client_gallery_photos
       (id, gallery_id, storage_key, filename, width, height,
        file_size, sort_order, created_at)
     VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?)`
  ).bind(
    photoId, galleryId, storageKey, safeFilename,
    width, height, file.size, sortOrder, now
  ).run();

Required compensation:

  let uploaded = false;
  try {
    await env.PHOTOS.put(storageKey, file.stream(), metadata);
    uploaded = true;
    await insertPhotoRow();
  } catch (error) {
    if (uploaded) await env.PHOTOS.delete(storageKey).catch(() => undefined);
    throw error;

Additional requirements:

  - Validate all files before writing the first object.
  - Limit files per request.
  - Limit aggregate bytes per request and per account.
  - Derive extension from validated content, not the original filename.
  - Do not swallow R2/D1 consistency errors.

AP-004 [P0] CLIENT GALLERY PHOTOS CANNOT BE SERVED

Evidence:

  Client gallery APIs return:
    /api/media/photos/<client storage key>

  functions/api/media/[[path]].ts recognizes event photos/videos, album photos,
  and photographer portfolio records. It never resolves client_gallery_photos.

Impact:

  A successful client gallery upload still displays as 403/broken image.

Recommended fix:

  Add an explicit media policy resolver. A client photo is public only when:

  - client site is published and not deleted
  - gallery is not deleted
  - photo belongs to that gallery

  Draft-site owners may access through an authenticated owner path or a scoped
  preview signature.

Sample policy query:

  SELECT cgp.storage_key, cs.photographer_id, cs.published
  FROM client_gallery_photos cgp
  JOIN client_galleries cg ON cg.id = cgp.gallery_id
  JOIN client_sites cs ON cs.id = cg.client_site_id
  WHERE cgp.storage_key = ?
    AND cg.deleted_at IS NULL
    AND cs.deleted_at IS NULL

Decision:

  public access = cs.published = 1
  owner access  = authenticated user ID equals cs.photographer_id

Do not expose storageKey in the public JSON response. Return a URL only.

Acceptance tests:

  - Published client photo returns 200.
  - Draft client photo returns 403 anonymously.
  - Owner preview returns 200.
  - Deleted gallery/site photo returns 404.

AP-009 [P1] GATE TOKENS CAN BE TREATED AS REAL USERS

All client-management endpoints call requireAuth but do not require a real user
session, approved photographer/account, role, entitlement, or quota. The current
gate identity uses a synthetic "viewer" user ID/role.

Impact:

  A gate token can potentially reach authenticated account actions, create
  client sites, and consume storage. A normal viewer account can also create
  an unlimited number of client sites unless another layer happens to block it.

Recommended model:

  type AuthPayload =
    | { typ: "user"; userId: string; role: Role; sessionId: string }
    | { typ: "gate"; spaceId: string; role: "viewer" };

  export function requireUserSession(
    auth: AuthPayload
  ): Extract<AuthPayload, { typ: "user" }> | Response {
    if (auth.typ !== "user") {
      return json({ error: "User account required" }, 403);
    }
    return auth;
  }

For client management, also query an account entitlement:

  - approved photographer, platform owner, or explicit site-builder plan
  - account not disabled/deleted
  - within site/storage limits

Acceptance tests:

  - Gate token gets 403 for every client write endpoint.
  - Real unauthorized viewer gets 403.
  - Entitled owner succeeds.
  - Another owner gets 403 for site/gallery/blog IDs.

Evidence:

  Wrangler reports no migrations folder configured/found.
  wrangler.toml has no migrations_dir.

Impact:

AP-011 [P1] SLUG VALIDATION ALLOWS ROUTE COLLISIONS

Current validation accepts any lowercase letters/numbers/hyphens.

Examples that conflict with application routes:

  login, signup, dashboard, clients, site, blog, citysite, albums,
  photographers, inspiration, admin, s, api

The generic "/:slug" route also appears before the final "*" route and can
turn unrelated unknown URLs into a misleading client/photographer lookup.

Recommended fix:

  const RESERVED_SLUGS = new Set([
    "admin", "albums", "api", "blog", "citysite", "clients", "dashboard",
    "inspiration", "login", "photographers", "s", "signup", "site",
  ]);

  function validateSlug(value: string): string {
    const slug = value.trim().toLowerCase();
    if (!/^[a-z0-9](?:[a-z0-9-]{1,61}[a-z0-9])?$/.test(slug)) {
      throw new ValidationError("Slug must be 3-63 characters");
    }
    if (RESERVED_SLUGS.has(slug)) {
      throw new ValidationError("This slug is reserved");
    }
    return slug;
  }

Soft-delete policy must be explicit:

  - Either keep slugs permanently reserved, and query all rows for uniqueness.
  - Or release slugs by moving deleted rows to a tombstone slug.

Acceptance tests:

  - Reserved and malformed slugs are rejected.
  - Mixed-case input is normalized before uniqueness lookup.
  - Unknown app URL reaches the true NotFoundPage.
  - Deleted-site slug policy is deterministic.


AP-012 [P1] CUSTOM DOMAINS ARE STORED BUT NOT IMPLEMENTED

Current behavior:

  - client_sites.custom_domain accepts arbitrary text
  - middleware/domain lookup only resolves spaces
  - frontend CustomDomainRedirect only calls /api/spaces/by-domain
  - no DNS ownership verification
  - no uniqueness/normalization workflow
  - no Cloudflare custom-hostname/domain provisioning

Impact:

  The UI promises a feature that cannot route. It can also store domains the
  user does not own.

Immediate recommendation:

  Hide/disable client custom-domain UI and label it "Coming soon" until the full
  workflow exists. Do not pretend manual text entry connects a domain.

Full implementation:

  1. Normalize with URL/hostname parsing; reject paths, ports, and IPs.
  2. Store lowercase ASCII hostname.
  3. Create pending domain record, not active custom_domain.
  4. Give user a DNS TXT/CNAME verification value.
  5. Verify ownership server-side.
  6. Provision through an operator-controlled Cloudflare workflow.
  7. Route host to client_sites as well as spaces.
  8. Handle removal, conflicts, certificate status, and retries.

At scale, evaluate Cloudflare for SaaS/custom hostnames rather than manually
adding every customer hostname to a Pages project.


AP-013 [P1] CLIENT DELETE LEAVES PHOTOS AND STORAGE FOREVER

Deleting a client site soft-deletes the site, blogs, and galleries, but leaves:

  - client_gallery_photos rows
  - R2 objects

Gallery deletion attempts R2 deletion, swallows errors, then removes D1 rows.
This can permanently lose the only retry record while leaving storage orphaned.

Recommended fix:

  Use a deletion job/tombstone:

  deletion_jobs(
    id, resource_type, resource_id, storage_key,
    status, attempts, last_error, created_at, completed_at
  )

  - Mark logical records deleted first.
  - Enqueue one job per storage object.
  - Delete through a Queue/consumer or scheduled retry.
  - Only remove/tombstone the photo row after confirmed storage deletion.
  - Keep a dead-letter state for operator repair.

AP-017 [P1] SIGNED MEDIA URLS BYPASS RESOURCE POLICY

Current signed URL verification is performed before the database policy lookup.
A valid signature authorizes the storage key even if the requested key is not
the intended album/resource. The signature binds only key and expiry.

Additional concerns:

  - dedicated media secret is not required
  - direct string comparison remains in signed-url verification
  - album ownership/audience is not included in the signature

Recommended token claims:

  interface MediaGrant {
    v: 1;
    key: string;
    kind: "photo" | "video";
    audience: "public-site" | "album-share" | "owner-preview";
    resourceId: string;
    expires: number;
  }

Sign canonical JSON or a fixed canonical field string. Verify:

  - HMAC using a dedicated MEDIA_SIGNING_SECRET
  - expiry
  - kind
  - storage key
  - resource ID
  - current resource still active/allowed

Use crypto.subtle.verify for constant-time verification.


AP-018 [P1] ALBUM RAW-KEY ACCESS IS TOO PERMISSIVE

Media lookup allows album_photos by storage key. Password-protected albums rely
on signatures, but unprotected albums can be fetched by key without proving the
share token. Album deleted_at and expiry are not consistently checked.

Recommended fix:

  - Require a scoped album grant for all nonpublic album media.
  - Check albums.deleted_at IS NULL and expires_at at access time.
  - Do not treat an unguessable R2 key as authorization.
  - Fail album view if no dedicated signing secret exists; never sign with "".


AP-019 [P1] SOFT-DELETED EVENTS CAN STILL BE READ/UPDATED DIRECTLY

functions/api/events/[id].ts selects:

  SELECT * FROM events WHERE id = ?

for GET, PUT, and DELETE without deleted_at IS NULL.

Recommended fix:

  Apply active-event filtering in one shared loader. Return 404 for deleted
  events. Repeat the active predicate in event-access and upload authorization.


AP-020 [P1] EVENT STORAGE CLEANUP CAN TIME OUT

Event deletion loops through R2 copy/delete work inside the request and swallows
cleanup errors. Large events can exceed the request lifetime and leave partial
results.

Recommended fix:

  - Logical delete and enqueue cleanup jobs in the request.
  - Process objects through Cloudflare Queues or a retryable workflow.
  - Record per-object status.
  - Use waitUntil only for best-effort noncritical work, not the sole deletion
    guarantee.


AP-021 [P1] PASSWORD RESET EXPIRY USES MIXED DATE FORMATS

forgot-password stores JavaScript ISO strings such as:

  2026-06-24T...Z

reset-password compares them lexicographically against SQLite datetime('now'):

  2026-06-24 ...

Those formats are not safely comparable as text.

Recommended fix:

  Store Unix epoch seconds consistently:

  const expiresAt = Math.floor(Date.now() / 1000) + 3600;

  SELECT ... WHERE token = ? AND expires_at_epoch > unixepoch()

Also revoke active sessions/JWT session records after a password reset.


AP-022 [P1] INVITATIONS TO NEW EMAILS ARE DEAD LINKS

connections/index.ts says account creation happens on magic-login acceptance.
magic-login actually returns "Account not found" when the email has no user.

Choose and implement one policy:

  A. Invite flow creates a pending signup and completes account creation after
     email ownership is proven.
  B. Invite flow redirects new recipients to signup while preserving a
     single-use invite token.

Do not silently promise automatic account creation without code to perform it.
Reduce the current seven-day magic-link lifetime and record used_at/revoked_at.


AP-023 [P1] JWT LOGOUT/REVOCATION REMAINS INCOMPLETE

JWT lifetime is improved to 24 hours, but:

  - logout is client-side only
  - no session ID/revocation table
  - role/account changes do not immediately revoke existing tokens
  - token is stored in localStorage

Recommended staged fix:

  1. Add sessions table and sid claim.
  2. Check session active state for sensitive writes.
  3. Revoke sessions on password reset, account disable, and role downgrade.
  4. Prefer Secure, HttpOnly, SameSite cookies when the CSRF model is ready.


AP-024 [P1] ADMIN USER DELETE CAN VIOLATE OWNERSHIP/FKS

The prior space-wipe defect was fixed, but hard-deleting users can still collide
with spaces.owner_id, uploads, and other ownership records. It also lacks robust
self/last-platform-owner protections.

Recommendation:

  - Soft-disable users by default.
  - Transfer or explicitly resolve owned resources.
  - Prevent deleting/demoting self and the last platform owner.
  - Audit every admin mutation.
  - Use a transaction/batch for membership and role changes.


AP-025 [P1] EXISTING USER CANNOT BE ADDED TO ANOTHER SPACE CLEANLY

Space-member POST attempts to insert a new global user. Because users.email is
unique, inviting an existing account can produce a 500.

Fix:

  - Normalize email.
  - Find existing user.
  - Insert space_members for that user if not already present.
  - Otherwise create a pending invitation, not a duplicate user.
  - Return 409 for existing membership, not 500.

9. PHASE 4 - TEMPLATE SYSTEM SAFETY AND MAINTAINABILITY
===============================================================================

AP-030 [P1] TEMPLATES PUBLISH FABRICATED CLAIMS AS DEFAULT CONTENT

Many generated templates contain fixed public claims, credentials, prices,
testimonials, revenue figures, client counts, health/safety assertions, and
professional qualifications.

Examples include childcare licensing/CPR/security claims, coaching credentials
and revenue claims, fixed medical/legal/business claims, and invented customer
testimonials.

Impact:

  A customer can publish false advertising or regulated-industry claims without
  realizing the text is fictional. This is a legal and trust risk, not merely a
  copywriting concern.

Required rule:

  Templates may contain obvious preview placeholders, but published sites must
  display only user-confirmed claims. Never silently publish credentials,
  ratings, client counts, testimonials, guarantees, prices, or safety claims.

Recommended content model:

  interface Claim {
    id: string;
    text: string;
    category:
      | "credential"
      | "testimonial"
      | "statistic"
      | "safety"
      | "price";
    confirmedByUser: boolean;
    sourceNote?: string;
  }

Published renderer:

  const visibleClaims = claims.filter((claim) => claim.confirmedByUser);

Add a pre-publish checklist requiring confirmation for regulated claims.


AP-031 [P1] PHOTOGRAPHER PROPS ARE REUSED FOR UNRELATED INDUSTRIES

The shared TemplateProps model is photography-oriented: portfolio, download
pricing, verified photographer, and onHire. It is used for doctors, lawyers,
childcare, finance, coaches, and other industries.

Symptoms:

  - "Hire Me" on professional/medical templates
  - photo-download price reused as service pricing
  - empty portfolio passed to non-photo sites
  - CTA often does nothing

Recommended architecture:

  interface BaseSiteProps {
    businessName: string;
    bio: string;
    theme: ThemeConfig;
    media: SiteMedia[];
    primaryAction: ActionConfig;
  }

  type IndustrySiteProps =
    | { industry: "photography"; portfolio: Photo[]; licensing: ... }
    | { industry: "health"; services: Service[]; appointment: ... }
    | { industry: "legal"; practices: PracticeArea[]; consultation: ... }
    | { industry: "childcare"; programs: Program[]; inquiry: ... };

Templates should consume a normalized view model, not reinterpret unrelated
photography fields.


AP-032 [P2] GOOGLE FONT IMPLEMENTATION CONFLICTS WITH CSP

Findings:

  - 262 of 263 template files reference Google Fonts
  - CSP permits self-hosted styles/fonts and Mapbox, not Google font origins
  - 156 templates append link elements
  - none of those templates remove the links on unmount
  - 106 templates use @import inside injected style blocks

Impact:

  Intended fonts are blocked by CSP, preview sessions accumulate links, and the
  implementation adds privacy/performance variability.

Recommended fix:

  Self-host an approved, small font set. Load it once in the application. Map
  template font choices to CSS variables. Remove per-template link injection
  and @import.

  :root {
    --font-display: "FrameNest Display", Georgia, serif;
    --font-body: "FrameNest Sans", system-ui, sans-serif;
  }


AP-033 [P2] TEMPLATE PICKER RENDERS TOO MUCH AT ONCE

Current assets:

  - 263 registered templates
  - 502 thumbnail files
  - approximately 40.2 MB of template thumbnails

TemplatePicker creates all template cards in the DOM. Lazy images reduce initial
network use but do not solve DOM/search/usability cost, and scrolling can still
download a large asset set.

Recommended fix:

  - search by name/industry
  - category filter
  - show 12-24 results per page
  - virtualize or paginate
  - remove obsolete duplicate thumbnail files
  - generate modern AVIF/WebP thumbnails at fixed dimensions
  - add width/height and meaningful alt text


AP-034 [P2] EACH TEMPLATE INJECTS LARGE LOCAL CSS

All 263 templates inject style tags, creating duplicated CSS patterns and making
visual maintenance expensive. One template uses dangerouslySetInnerHTML for
static CSS, contrary to repository guidance.

Recommended direction:

  - shared template primitives: header, hero, service grid, testimonial,
    contact section, gallery, footer
  - shared design tokens and CSS modules
  - template manifests that select layout/tokens
  - no dangerouslySetInnerHTML

Do this after functional contracts are fixed. A mass template rewrite before P0
is complete would increase risk.


AP-035 [P2] TEMPLATE INTERACTIONS ARE OFTEN DECORATIVE

Many navigation labels are spans instead of links/buttons. Some forms and CTAs
have no submit/action behavior. ClientSitePage currently provides no-op onHire.

Acceptance standard:

  - Every interactive-looking control is a semantic button/link/input.
  - Keyboard focus is visible.
  - Navigation changes URL or scrolls to a real section.
  - Form submit has validation, success, and error behavior.
  - No button is shipped with an empty handler.


AP-036 [P2] NO VISUAL REGRESSION COVERAGE FOR 263 TEMPLATES

Recommended test system:

  - render every template from a deterministic fixture
  - desktop and mobile screenshot
  - detect runtime errors and horizontal overflow
  - sample accessibility scan for every template
  - full screenshot diff for a representative template per layout family

Do not require manual inspection of 526 screenshots on every change. Group
templates by shared layout once the component model is normalized.
