2. High: album upload accepts weakly validated files and has no size/count cap

Impact
The public album upload endpoint can store arbitrary or oversized content if the browser-provided MIME type is missing or misleading. This risks storage abuse, unexpected downloads, and operational cost.

Evidence
- functions/api/albums/[id]/photos.ts calls await request.formData() and stores files in R2.
- It uses validateUploadContent(file), but validateUploadContent mostly rejects known dangerous signatures and accepts files when type is empty or not image/* unless the file starts with HTML/SVG patterns.
- No explicit max file size, max file count, total request size, or allowlist MIME check is enforced in this route.

Suggested shared upload policy

```ts
// functions/lib/upload-policy.ts
export const PHOTO_MIME_TYPES = new Set([
  "image/jpeg",
  "image/png",
  "image/webp",
  "image/gif"
]);

export const VIDEO_MIME_TYPES = new Set([
  "video/mp4",
  "video/webm",
  "video/quicktime"
]);

export const MAX_ALBUM_PHOTO_BYTES = 15 * 1024 * 1024;
export const MAX_ALBUM_PHOTO_COUNT = 20;
export const MAX_ALBUM_UPLOAD_BYTES = MAX_ALBUM_PHOTO_BYTES * MAX_ALBUM_PHOTO_COUNT;

export function safeImageExtension(file: File): string {
  switch (file.type) {
    case "image/jpeg": return "jpg";
    case "image/png": return "png";
    case "image/webp": return "webp";
    case "image/gif": return "gif";
    default: return "bin";
  }
}

export function rejectOversizedRequest(request: Request, maxBytes: number): Response | null {
  const rawLength = request.headers.get("content-length");
  if (!rawLength) return null;

  const contentLength = Number(rawLength);
  if (!Number.isFinite(contentLength) || contentLength < 0) {
    return new Response(JSON.stringify({ error: "Invalid Content-Length" }), {
      status: 400,
      headers: { "Content-Type": "application/json" }
    });
  }

  if (contentLength > maxBytes) {
    return new Response(JSON.stringify({ error: "Upload too large" }), {
      status: 413,
      headers: { "Content-Type": "application/json" }
    });
  }

  return null;
}
```

Suggested album upload route checks

```ts
const tooLarge = rejectOversizedRequest(context.request, MAX_ALBUM_UPLOAD_BYTES);
if (tooLarge) return tooLarge;

const formData = await context.request.formData();
const files = formData.getAll("photos").filter((entry): entry is File => entry instanceof File);

if (files.length === 0) {
  return Response.json({ error: "No photos provided" }, { status: 400 });
}

if (files.length > MAX_ALBUM_PHOTO_COUNT) {
  return Response.json({ error: "Too many photos in one request" }, { status: 413 });
}

for (const file of files) {
  if (!PHOTO_MIME_TYPES.has(file.type)) {
    return Response.json({ error: `Unsupported image type: ${file.type || "unknown"}` }, { status: 400 });
  }

  if (file.size <= 0 || file.size > MAX_ALBUM_PHOTO_BYTES) {
    return Response.json({ error: "Photo file is too large" }, { status: 413 });
  }

  const contentCheck = await validateUploadContent(file);
  if (!contentCheck.valid) {
    return Response.json({ error: contentCheck.error }, { status: 400 });
  }
}
```

Also update validateUploadContent so it fails closed when MIME is missing for upload routes that require a photo or video.

3. High: authenticated photo/video uploads parse full form data before request size checks

Impact
The app reads the entire multipart body before deciding whether it is too large. This can waste memory and CPU at the edge. Cloudflare guidance is to enforce limits before consuming large bodies.

Evidence
- functions/api/photos/upload.ts calls await context.request.formData() before checking file sizes.
- functions/api/videos/upload.ts calls await context.request.formData() before checking file sizes.
- Neither route caps file count or total size.

Suggested fix
- Check Content-Length before formData().
- Enforce max count.
- Enforce per-file and total bytes after parsing.
- For larger production uploads, switch to direct-to-R2 signed upload URLs or multipart/resumable upload flow and keep the Function as the metadata/finalize endpoint.

Suggested helper

```ts
function sumFileBytes(files: File[]) {
  return files.reduce((sum, file) => sum + file.size, 0);
}

function requireFileBatch(files: File[], options: {
  maxCount: number;
  maxEachBytes: number;
  maxTotalBytes: number;
  allowedTypes: Set<string>;
}) {
  if (files.length === 0) return "No files provided";
  if (files.length > options.maxCount) return "Too many files";

  const total = sumFileBytes(files);
  if (total > options.maxTotalBytes) return "Upload batch is too large";

  for (const file of files) {
    if (!options.allowedTypes.has(file.type)) return `Unsupported file type: ${file.type || "unknown"}`;
    if (file.size <= 0 || file.size > options.maxEachBytes) return "One or more files are too large";
  }

  return null;
}
```

4. High: signed media URLs should sign the media kind and use constant-time comparison

Impact
The current signature signs storageKey and expires, but not kind. If the same storage key exists in both buckets, a signature may be replayed across photo/video routes. The comparison also uses direct string equality, which is not ideal for secret signatures.

Evidence
- functions/lib/signed-url.ts signs `${storageKey}:${expires}`.
- functions/lib/signed-url.ts verifies with `expected === signature`.
- functions/api/media/[[path]].ts bypasses DB authorization when a signed URL is valid.

Suggested signed URL implementation

```ts
async function hmacHex(secret: string, data: string): Promise<string> {
  const key = await crypto.subtle.importKey(
    "raw",
    new TextEncoder().encode(secret),
    { name: "HMAC", hash: "SHA-256" },
    false,
    ["sign"]
  );

  const signature = await crypto.subtle.sign("HMAC", key, new TextEncoder().encode(data));
  return Array.from(new Uint8Array(signature))
    .map((byte) => byte.toString(16).padStart(2, "0"))
    .join("");
}

function constantTimeEqualHex(a: string, b: string): boolean {
  const maxLength = Math.max(a.length, b.length);
  let diff = a.length ^ b.length;

  for (let i = 0; i < maxLength; i++) {
    diff |= (a.charCodeAt(i) || 0) ^ (b.charCodeAt(i) || 0);
  }

  return diff === 0;
}

function mediaSigningPayload(kind: "photos" | "videos", storageKey: string, expires: number) {
  return `v1:${kind}:${storageKey}:${expires}`;
}

export async function signMediaUrl(
  storageKey: string,
  kind: "photos" | "videos",
  secret: string,
  ttlSeconds = 900
): Promise<string> {
  const expires = Math.floor(Date.now() / 1000) + ttlSeconds;
  const signature = await hmacHex(secret, mediaSigningPayload(kind, storageKey, expires));
  return `/api/media/${kind}/${encodeURIComponent(storageKey)}?expires=${expires}&signature=${signature}`;
}

export async function verifyMediaSignature(
  storageKey: string,
  kind: "photos" | "videos",
  expires: string | null,
  signature: string | null,
  secret: string
): Promise<boolean> {
  if (!expires || !signature) return false;

  const expiresAt = Number(expires);
  if (!Number.isFinite(expiresAt) || expiresAt < Math.floor(Date.now() / 1000)) return false;

  const expected = await hmacHex(secret, mediaSigningPayload(kind, storageKey, expiresAt));
  return constantTimeEqualHex(expected, signature);
}
```

Recommended extra hardening
- Keep the TTL short, for example 10 to 15 minutes for private media.
- Consider checking the database row still exists and is not deleted even when the signature is valid.
- Add tests that a photo signature cannot access the videos bucket and vice versa.

5. High: custom domains need validation, uniqueness, and ownership verification

Impact
Users can currently submit arbitrary custom_domain values for spaces/client sites. Without DNS verification, a user can claim a domain they do not control. Without normalization and uniqueness checks, the same hostname can be stored in multiple forms.

Evidence
- functions/api/spaces/[slug].ts accepts custom_domain from the request body.
- functions/api/clients/[id].ts accepts custom_domain from the request body.
- db/schema.sql does not show a unique index for client_sites.custom_domain.

Suggested hostname normalization

```ts
export function normalizeCustomDomain(input: unknown): string | null {
  if (typeof input !== "string") return null;

  const value = input.trim();
  if (!value) return null;

  let host: string;
  try {
    const url = value.includes("://") ? new URL(value) : new URL(`https://${value}`);
    host = url.hostname.toLowerCase();
  } catch {
    return null;
  }

  const labels = host.split(".");
  const valid = labels.length >= 2 && labels.every((label) =>
    label.length > 0 &&
    label.length <= 63 &&
    !label.startsWith("-") &&
    !label.endsWith("-") &&
    /^[a-z0-9-]+$/.test(label)
  );

  if (!valid) return null;
  if (host === "framenest.photos" || host.endsWith(".framenest.photos")) return null;

  return host;
}
```

Recommended schema changes

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_spaces_custom_domain_unique
  ON spaces(custom_domain)
  WHERE custom_domain IS NOT NULL;

CREATE UNIQUE INDEX IF NOT EXISTS idx_client_sites_custom_domain_unique
  ON client_sites(custom_domain)
  WHERE custom_domain IS NOT NULL;

ALTER TABLE spaces ADD COLUMN custom_domain_verified_at TEXT;
ALTER TABLE spaces ADD COLUMN custom_domain_verification_token TEXT;

ALTER TABLE client_sites ADD COLUMN custom_domain_verified_at TEXT;
ALTER TABLE client_sites ADD COLUMN custom_domain_verification_token TEXT;
```

Recommended workflow
- Save normalized hostname plus a random verification token.
- Ask the owner to add a TXT record like `_framenest.<domain>`.
- Only route traffic for a custom domain after the TXT record validates.
- Return 409 if another space/client site already owns the hostname.

Security hardening checklist

Authentication and authorization
- Add token_version session revocation.
- Audit all endpoints for requireAuth plus requireRole/ownership checks.
- Add tests that viewer/staff/page_admin/platform_owner boundaries match the intended matrix.
- Consider HttpOnly cookie auth with CSRF protection.
- Add "log out all devices" for account settings and platform owner tools.

Uploads and media
- Fail closed on unknown MIME types.
- Enforce request Content-Length before reading multipart bodies.
- Add max count and total batch bytes.
- Re-encode images or generate safe derivatives before serving thumbnails/previews.
- Store user-visible filenames separately from storage keys.
- Keep media response headers strict:

```txt
X-Content-Type-Options: nosniff
Content-Disposition: inline; filename="<safe-name>"
Cache-Control: private, max-age=300
```

Use attachment disposition for unknown or untrusted types.

Rate limiting and abuse protection
- Rate limit by user ID plus IP for authenticated routes.
- Add per-space/per-album upload quotas.
- Return Retry-After on 429 responses.
- Add Cloudflare Turnstile to signup, login, forgot password, public album upload, booking inquiry, and gate-key forms where abuse is likely.

Data protection
- Add audit_logs/activity_logs for admin actions: role change, custom domain change, delete/restore, export, password reset.
- Add soft-delete restore workflow and retention policy for spaces/events/photos/videos.
- Add D1 backup before deploy and document restore steps.

Cloudflare-specific improvements
- Use ctx.waitUntil for non-critical background tasks.
- Keep compatibility_date current after preview testing.
- Generate Cloudflare binding types.
- Enable Workers/Pages observability.
- Prefer bindings for D1/R2/KV/secrets. Do not use account API tokens inside runtime code unless absolutely necessary.
- Avoid global mutable request state. Global cached clients/maps are fine only when they do not store per-request/user data.
