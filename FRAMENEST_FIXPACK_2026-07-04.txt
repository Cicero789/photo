ANSWER 3 — FINDINGS NOT INCLUDED IN THE v2 REPORT
================================================================================
Everything below was observed during the line-by-line review but cut for
brevity or because it "works today". Ordered roughly by how much I'd care.

C-1  GATE VIEWERS SHARE ONE GLOBAL IDENTITY (closest to a real bug)
     functions/api/auth/gate.ts:11 signs every gate token with
     userId:"viewer" — the SAME literal for every visitor of every space.
     Consequences on any endpoint keyed by userId:
     - functions/api/inspiration/[id]/love.ts: one gate viewer's love
       INSERTs (inspiration_id,'viewer') — every other gate viewer worldwide
       now sees that pin as loved, and any of them can un-love it.
     - functions/api/inspiration.ts:32-44 (POST): a gate viewer submitting a
       pin stores user_id='viewer'; the GET at :11 does `JOIN users u ON
       i.user_id = u.id`, and since no users row has id 'viewer', the pin
       silently VANISHES from the map. Data loss with no error.
     - functions/api/notifications.ts: GET/PUT on user_id 'viewer' — shared
       notification state across all gate viewers (currently benign because
       nothing writes activity_log for 'viewer', but it's a trap).
     This also brushes against repo rule DO-NOT #9 ("never allow a gate
     token to act as a real user account").
     FIX OPTIONS (pick one):
     (a) Block role==='viewer' from identity-keyed writes: in
         inspiration/[id]/love.ts and inspiration.ts POST add
         `if (a.role === "viewer") return json({ error: "Sign in to do that" }, 403);`
     (b) Or sign gate tokens with a per-space synthetic id in gate.ts:11:
         userId: `gate:${space.id}` — scoped, still not a real account.
     (a) is safer and matches the product model (viewers browse, not write).

C-2  NO-PASSWORD ALBUM PHOTOS ARE KEY-SECRECY-ONLY
     functions/api/media/[[path]].ts:74-82: if a storage key belongs to an
     album WITHOUT a password, the file is served to ANYONE who knows the
     key — the share token is never checked on the media route. Keys contain
     UUIDs so they're not guessable, but every viewer of a shared album can
     hotlink files forever, and revoking a share link does not revoke media
     access. Works as designed today; tighten later by requiring a signed
     URL for album media (the album view endpoint already signs URLs when a
     password is set — extend that to all albums).

C-3  LOGOUT IS A SERVER-SIDE NO-OP / NO TOKEN REVOCATION
     functions/api/auth/logout.ts returns {success:true} and nothing else;
     JWTs carry a jti (functions/lib/jwt.ts:24) but no revocation list
     exists, so a stolen token stays valid for the full 24h regardless of
     logout or password change. Related: the token lives in localStorage
     (src/lib/api.ts:7), which is XSS-readable — acceptable now that the
     blog XSS is fixed and CSP is strict, but it's the reason XSS findings
     here are always account-takeover severity. Cheap upgrade later: store
     a `tokenVersion` per user, embed it in the JWT, bump on password change.

C-4  DUPLICATED MAPBOX CDN LOADER WITH A RACE
     src/components/map/EventMap.tsx:35-48 injects the mapbox-gl <script>
     with no shared-promise guard; InspirationMapPage.tsx:21-36 has a proper
     `mapboxPromise` singleton; SpaceEventMap.tsx:24-33 is a third copy.
     Two maps mounting near-simultaneously (e.g. EventDetailPage +
     SpaceEventMap) can double-inject the script or read window.mapboxgl
     while it's still undefined. When doing D-8, consolidate all three into
     one src/lib/mapbox.ts loader (token + loadMapbox() promise).

C-5  ADMIN COMMERCE "PLATFORM FEES" IS AN INVENTED NUMBER
     src/pages/admin/AdminCommercePage.tsx:88 computes
     `platformFees: Math.round(totalRevenue * 0.05)` — there is no 5% fee
     recorded anywhere in the orders schema; this is a guess rendered as
     fact. Label it "(est.)" in the UI or derive from real Stripe
     application-fee data when Connect fees are implemented. Same page :90:
     verifiedSubscribers is hardcoded 0 — either fetch photographersVerified
     from /admin/stats or drop the tile.

C-6  "VERIFIED" vs "APPROVED" SEMANTICS ARE CONFLATED
     src/pages/admin/AdminDashboardPage.tsx:118-119 warns "No verified
     photographers — directory will appear empty" based on
     photographersVerified (the paid badge), but directory eligibility is

     directory and make the alert + filters agree, or the dashboard will

C-7  UPLOADS TRUST CLIENT METADATA BLINDLY
     functions/api/photos/upload.ts:14,19 stores client-supplied width/
     height/latitude/longitude/takenAt verbatim (0/null defaults). Bogus
     dimensions break aspect-ratio layouts; spoofed geotags put pins
     anywhere on space maps. Low risk (staff-only upload), but validate
     ranges (lat -90..90, lng -180..180, width/height 0..20000) when handy.

C-8  MULTI-FILE UPLOADS ARE FULLY SEQUENTIAL
     functions/api/photos/upload.ts:16-22 awaits R2 put + D1 insert per file
     in a loop; a 20-photo upload pays 40 sequential round-trips and risks
     the 30s Workers CPU/duration budget. Batch the D1 inserts (db.batch)
     and Promise.all the R2 puts in chunks of ~5. Same pattern in
     videos/upload.ts:16-21.

C-9  EVENT DELETE DOES R2 TRASH-COPY IN-REQUEST

     trash/ and deletes originals synchronously before responding; a
     500-photo event will time out (DB soft-delete already happened, so
     state stays correct — but the user sees an error). Wrap the R2 loop in
     context.waitUntil (same technique as D-11).

C-10 spaces PUT LACKS INPUT VALIDATION
